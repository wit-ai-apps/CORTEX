## Phase2：比較・正式意見

### 比較対象

| 案 | 概要 |
|---|---|
| A | ローカル用ダッシュボード（静的HTML） |
| B | CLI `cortex-status` 相当の要約表示 |
| C | `/code` に最小プロジェクト枠のみ |
| D | GitHub Actions で MD 命名・ヘッダ検査 |
| E | ローカル限定の小さなAPI（メタ読み取りのみ） |

---

### Claudeの推し：案D

**理由：**

CORTEXの本質は「各AIが正しいフォーマットで書き込み、なべさんが判断する」ことだ。

```
フォーマット違反が混入した瞬間
  → なべさんの判断コストが上がる
  → ディスカッションの品質が下がる
  → CORTEXの価値が損なわれる
```

案A・Bは「現状を見る」道具。  
案Dは「正しさを守る」道具。

**最初に守る仕組みを作ると、後の全議論の品質が自動で上がる。**  
これがCORTEXを最短で軌道に乗せる方法だ。

---

### 各案への評価

**案A（ダッシュボード）への異議：**

最初の機能としては装飾的すぎる。  
CORTEXにディスカッションが蓄積されてから作るべきものだ。  
今は「見る」ものより「動かす」ものが必要。

**案B（CLI）への異議：**

再現性は高いが、CLIは「人間が叩く」前提。  
CORTEXは最終的に自動化を目指すため、  
人間が叩かなくても動くDの方が方向性が正しい。

**案C（枠のみ）への異議：**

枠だけでは何も検証できない。  
「動いた」という実感が得られず、モチベーションが下がるリスクがある。

**案E（API）への異議：**

初手としてスコープが広がりすぎる。  
秘密情報・トークン管理の問題が発生するリスクがある。

**案D（GitHub Actions）の弱点：**

正直に申告する。

```
- Actions設定に初期コストがかかる
- 各AIがActionsの仕様を理解する必要がある
- 設定ミスで誤検知が出る可能性がある
```

ただしこれらは**一度設定すれば消える**コストだ。  
A・B・Eの「継続的な運用コスト」とは性質が違う。
