## Phase1：提案（テーマ）

**ISSUE:** CORTEXで最初に実装すべき機能は何か？

### Cursor による叩き台（提案）

**提案内容:** `/code` に **「議論・決定・履歴を一覧するローカル用ダッシュボード（静的HTMLまたは単一バイナリ）」** を置く。

**理由:** リポジトリを開いただけでは Phase の進みが見えにくい。ファイルは増えるため、**現状スナップショット**（discussion 件数・decisions 最新・history の有無）を一画面で把握できると、なべさんと各 AI の同期コストが下がる。

**想定リスク:** 実装が肥大化しやすい → **MVP は「Markdown の一覧と最終更新日時のみ」**に限定する。

## Phase2：比較（視点別の追記）

### Claude視点（設計・原因分析の参考）

**比較対象案:**  
A: 上記ダッシュボード  
B: **CLI `cortex-status`**（引数なしで要約表示）  
C: **まずは `/code` に最小アプリの枠のみ**（例: Hello + CI のみ）で運用ルールを固める

**強み / 弱み（要約）:**  
- A は可視化が強いが、ホストやブラウザ前提が増える。  
- B は自動化・CI と相性がよく、**再現性**が高い。  
- C は議論と並行しやすいが、「CORTEX らしさ」はテンプレ整備に依存する。

**自分の案（Claude としての推し）との差:** 原因分析の観点では **B** がログ取り込みと相性がよい。  
**採用する場合の注意点:** どの案も **スコープ上限**（1スプリント相当）を README か discussion に先に書くとブレにくい。

---

### Gemini視点（比較・別視点の参考）

**比較対象:** A（ダッシュボード）／B（CLI）／D（**GitHub Actions で discussion のフロントマター検査**）

**強み:** D は「命名規則 v1.0.0 の自動チェック」に直結し、**人間のレビュー負荷**を下げられる。  
**弱み:** D 単体ではユーザー向け「機能」として見えにくい。A/B は体感価値が出やすい。

**自分の案との差:** 「最初の機能」を **開発者体験（DX）** と定義するなら D+A のハイブリッドもあり。  
**採用時の注意:** Actions はリポジトリ設定とトークン権限の理解が必要。

---

### Cursor視点（実装の参考）

**比較対象:** A / B / **E: `/code` に小さな REST API（health + リポメタ読み取り）`**

**強み:** E は後からダッシュボードや CLI のバックエンドに流用できる。  
**弱み:** 秘密情報・トークンを触ると危険。**ローカルファイルのみ読む**範囲に限定すべき。

**自分の案（叩き台 A）との差:** E は拡張性が高いが **初速は遅い**。初回は **B か A の静的版**が安全。  
**採用する場合の注意点:** `node` / `go` / `python` のどれにするか、**README で一つに固定**してから着手する。
