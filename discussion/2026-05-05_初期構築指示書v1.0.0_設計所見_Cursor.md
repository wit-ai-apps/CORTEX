## Phase1：提案

### 提案内容

1. **`/history` の役割を README に一言定義する**  
   Git のコミット履歴と二重にならないよう、「マイルストーン要約」「ADR 置き場」など用途を明示する。

2. **Phase5・6 の記録場所をテンプレまたは README で明示する**  
   Discussion テンプレは Phase1〜4 まで。Phase5（なべさん選択）・Phase6（決定保存）は `/decisions` で完結する旨を `_template.md` 先頭に 1 行あると取り違えが減る（v1.1 または運用メモでも可）。

3. **受け入れ条件 1 と実行主体の切り分け**  
   GitHub リポジトリ作成はなべさん側、初期ファイル配置は Cursor 等、と一文書いておくと完了報告の齟齬を防げる。

4. **（中優先）Discussion ファイルの推奨命名**  
   例：`yyyy-mm-dd_課題スラッグ.md` を README または v1.1 で推奨。

5. **（中優先）`AI:` 欄の「Cursor」の定義**  
   IDE／エージェント基盤名とモデル名のどちらを必須にするか、用語を揃える。

6. **（中優先）Phase3 グループ表に「その他」または備考**  
   固定軸に当てはまらない課題への逃げ道。

### 理由

- v1.0.0 は最小構成として十分実行可能だが、上記は**運用で揉めやすい境界**（履歴・Phase・GitHub 主体）に焦点を当てたものである。
- 指示書の禁止事項（テンプレ不変更）を尊重するなら、**README 追記や v1.1、運用メモ**で吸収できる。

### 想定リスク

- README に追記する場合、なべさん・各 AI が最新版を参照する運用が必要。
- `_template.md` を v1.1 で変える場合、既存ログとの整合ルールが必要（テンプレは変えない方針とのトレードオフ）。

## Phase2：比較

### 比較対象

案名：**現行指示書（v1.0.0 最小構成）** vs **上記 README／v1.1 で境界を補強した構成**

### 強み

| 案 | 強み |
|---|---|
| 現行 | シンプル、禁止事項でブレにくい、受け入れ条件が明確。 |
| 補強 | `/history` や Phase の解釈ブレが減り、マルチ AI での衝突・取り違えが減る。 |

### 弱み

| 案 | 弱み |
|---|---|
| 現行 | `/history`・Phase5/6・GitHub 作成主体が未記載で、初回運用で質問が出やすい。 |
| 補強 | ドキュメントが増える分、更新忘れやバージョン差のリスク。 |

### 自分の案との差

本所見は「現行を否定せず、**追記で足りるものは追記**」が前提。大規模なフォルダ再設計は提案しない（指示書の禁止事項と整合）。

### 採用する場合の注意点

- テンプレ改変が禁止のままなら、**README または `discussion` 内の運用メモ**に寄せる。
- テンプレ改変を v1.1 で許容するなら、** SERIAL または VERSION** とセットで変更履歴を残すとよい。
